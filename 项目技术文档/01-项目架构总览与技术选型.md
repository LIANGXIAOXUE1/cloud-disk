# 项目架构总览与技术选型

## 一、这个项目是什么

私人聚合云盘，一句话概括：**一个界面管理你所有网盘的文件**。

你有阿里云盘、OneDrive、本地电脑、123云盘上的文件，不用逐个登录切换，在这个系统里就能全部看到、搜索、操作。上传文件时系统自动帮你决定存到哪个盘最合适。

使用场景：

* 在公司电脑上传文档 → 回家打开系统直接看到
* 百度网盘自动同步工作资料到本地 → 不装百度客户端就能搜
* 给朋友发个分享链接 + 提取码 → 对方直接下载

## 三、为什么是 SpringBoot 3 而不是 2

SpringBoot 3 有 3 个关键升级：

1. **JDK 17 基线**：不再支持 JDK 8/11，用到了 Records、Sealed Classes、Pattern Matching 等新特性，代码更简洁
2. **Jakarta EE 替换 javax**：命名空间从 `javax.*` 改为 `jakarta.*`，这是未来的标准
3. **Native 编译支持**：SpringBoot 3 + GraalVM 可以把 Java 编译成原生可执行文件，启动速度从秒级降到毫秒级

选 SpringBoot 3 不是因为追新，而是 SpringBoot 2 已经进入维护期，新项目直接上 3 是标准做法。

## 四、为什么 MyBatis-Plus 而不是 JPA

| 对比维度    | MyBatis-Plus     | JPA / Hibernate        |
| ------- | ---------------- | ---------------------- |
| SQL 控制力 | 强，手写 SQL 也可以自动生成 | 弱，HQL 自动翻译可能出问题        |
| 学习曲线    | 低，会写 SQL 就能用     | 高，要理解持久化上下文、懒加载        |
| 复杂查询    | Lambda 链式，IDE 提示 | Criteria API，难读难写      |
| 性能调优    | 直接优化 SQL         | Hibernate 生成的 SQL 不好控制 |

这个项目有复杂的分页搜索、多表关联（文件归属 + 回收站 + 分享），MyBatis-Plus 的 Lambda 查询不用担心字段名拼错，重构时 IDE 自动改名。

> 同类技术：JPA/Hibernate、JOOQ、原生 MyBatis。选 MyBatis-Plus 的核心原因：**国内最主流 + SQL 可控 + Lambda 防拼错**。

## 五、为什么 Redis + Redisson

**Redis 用它做什么：**

- Sa-Token 的 Session 存储（登录状态不在内存，重启不丢）
- 分片上传的进度元数据（上传到一半关浏览器，下次继续）
- 文件列表缓存（热点数据不用每次都查 MySQL）

**Redisson 用它做什么：**

Redisson 不是另一个 Redis，它是 **Redis 的 Java 客户端**——和 Jedis、Lettuce 是同类。但 Redisson 额外提供了：

- 分布式锁（`RLock`），自带"看门狗"自动续期，不用担心锁过期
- 限流器（`RRateLimiter`）
- 分布式集合（Map、List、Queue）

任务 6 的文件删除操作用到 Redisson 锁，防止两个人同时删同一个文件导致数据错乱。

> 同类技术：Jedis、Lettuce。选 Redisson 的核心原因：**不只是客户端，自带分布式锁和高层数据结构**。

## 六、为什么 RabbitMQ

这个项目有一个耗时操作：百度网盘转存。用户粘贴分享链接后，后端要去百度下载文件（可能几分钟到几十分钟），再上传到目标存储。如果同步执行，用户点了按钮就一直转圈等，体验极差。

**异步流程：**

```
用户点"转存" → 创建任务记录 → 扔消息到队列 → 立即返回"转存中"
                                ↓
                      RabbitMQ 消费者慢慢处理
                                ↓
                      下载 → 上传 → 更新任务状态
```

**为什么是 RabbitMQ 而不是 Kafka：**

| 对比    | RabbitMQ      | Kafka           |
| ----- | ------------- | --------------- |
| 适用场景  | 业务任务分发（秒~分钟级） | 大数据流（TB 级日志/埋点） |
| 部署复杂度 | 低，开箱即用        | 高，需 Zookeeper   |
| 消息确认  | 每条消息独立 ACK    | 偏移量批量确认         |
| 死信队列  | 原生支持          | 需手动实现           |

转存任务每分钟可能就几十条，RabbitMQ 完全够用，没必要上 Kafka。

> 同类技术：Kafka、RocketMQ、ActiveMQ。选 RabbitMQ 的核心原因：**可靠性高 + 死信队列 + 部署简单**。

## 七、为什么 Sa-Token

Java 鉴权框架主要有三个：

| 框架              | 复杂度 | 特点                    |
| --------------- | --- | --------------------- |
| Spring Security | 极高  | 功能全但配置繁琐，链式过滤器让人头大    |
| Shiro           | 中   | 比 Security 简单，但已多年不更新 |
| Sa-Token        | 极低  | 登录/踢人/权限一行注解，文档中文友好   |

**这个项目里 Sa-Token 做的事：**

1. `StpUtil.login(userId)` — 登录，自动生成 Token
2. `StpUtil.checkLogin()` — 路由拦截，没登录的请求直接打回
3. `StpUtil.getLoginIdAsLong()` — 任何地方都能拿到当前用户 ID

不用写 Token 生成逻辑、不用写过滤器、不用写 Redis 序列化——Sa-Token 全干了。

> 同类技术：Spring Security、Shiro、JWT 手写。选 Sa-Token 的核心原因：**代码量少 10 倍 + 中文文档 + 自带 Redis 集成**。

## 八、项目分层架构

```
┌─────────────────────────────────────────┐
│                 前端 Vue3                │
│           Axios → /api/*                │
└──────────────┬──────────────────────────┘
               │ cloud_disk_token (Header)
┌──────────────▼──────────────────────────┐
│          Controller 层                   │
│  接收请求、参数校验、返回 Result          │
│  @SaCheckLogin + @CheckFileOwner        │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│           Service 层                     │
│  业务逻辑、事务管理、同名检查             │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│         Repository 层                    │
│  MyBatis-Plus Mapper、实体类            │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│         MySQL / Redis / MinIO           │
│         数据持久化 + 缓存 + 文件存储     │
└─────────────────────────────────────────┘
```

每层只做自己的事，不越界：

- Controller：不写业务逻辑，只做参数接收和结果返回
- Service：不碰 HTTP 请求/响应，只处理数据和业务规则
- Repository：只负责数据库读写，不关心业务含义

这样做的好处：改了前端接口格式不影响 Service，改了 Service 逻辑不影响数据库查询。每个层可以独立测试、独立替换。

文件大小限制
------

**能传多大**取决于你的硬盘剩余空间，理论无上限。但前端有默认限制（比如 2GB），后端可调。

**技术实现**：分片上传 + 断点续传。100GB 文件切成 10MB 的块，一块块传，断了从上次传的块继续。
分享给朋友
-----

朋友**不需要登录**你的网站。分享流程：

你：选文件 → 生成分享链接（如 http://你的域名/share/abc123） + 4位提取码朋友：打开链接 → 输入提取码 → 直接下载

后端实现：

1. 生成唯一 share_code 存数据库，关联文件路径
2. 设置有效期（7天/1个月/永久）
3. 访问时校验提取码 + 是否过期
4. 直接返回文件流（FileInputStream → HttpServletResponse.getOutputStream()）

朋友全程看不到登录页面，就像奶牛快传那样。

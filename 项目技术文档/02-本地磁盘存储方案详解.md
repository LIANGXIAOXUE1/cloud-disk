# 本地磁盘存储方案详解

## 一、为什么不用 MinIO 了？

原计划用 MinIO（兼容 S3 协议的对象存储），但发现两个问题：

1. **多一个进程**：MinIO 要单独启动，项目多了个依赖，出问题要排查两个服务
2. **文件不直观**：文件存 MinIO 里，想直接打开看还得用客户端或 API 下载

改成**本地磁盘存储**，文件就在硬盘上，`D:\cloud-disk-files\` 目录下，想怎么看就怎么看。

> 同类技术：MinIO、阿里云 OSS、七牛云、腾讯云 COS。选本地磁盘的核心原因：**零外部依赖 + 文件直接可见**。

## 二、存储目录结构

```
D:\cloud-disk-files\                    ← 根目录，可在 application.yml 配置
    ├── upload\                         ← 用户上传的文件
    │    ├── 1\                         ← 用户 ID=1 的目录
    │    │    ├── 工作文档.pdf
    │    │    └── 照片\
    │    └── 2\
    │         └── ...
    ├── temp\                           ← 分片上传的临时块
    │    ├── upload_abc123\             ← 上传任务 ID
    │    │    ├── chunk_0
    │    │    ├── chunk_1
    │    │    └── ...
    │    └── ...
    └── baidu\                          ← 从百度网盘同步来的文件（任务11）
```

**为什么按用户 ID 分目录？**

- 权限隔离：用户 A 看不到用户 B 的文件（即使知道文件路径也访问不了）
- 清理方便：删用户时直接删整个目录
- 统计简单：`du -sh upload/1/` 就知道用户 1 用了多少空间

## 三、普通上传流程

```
前端选择文件 → POST /api/file/upload → 后端接收 MultipartFile
    ↓
生成唯一文件名（UUID + 扩展名）
    ↓
检查目标目录是否已有同名文件 → 有则自动加 (1)、(2) 后缀
    ↓
写入磁盘：D:\cloud-disk-files\upload\{userId}\{uuid}.{ext}
    ↓
写入数据库：file_info 表，storage_type="LOCAL"，file_path 存绝对路径
    ↓
返回 FileInfo JSON
```

**为什么用 UUID 而不是原文件名？**

1. 安全：防止文件名含 `../` 等路径穿越字符
2. 唯一：避免两个用户上传同名文件冲突
3. 简单：不用处理中文/特殊字符编码问题

原文件名存在 `file_info.file_name` 字段，展示时用这个。

## 四、分片上传（大文件断点续传）

**为什么需要分片上传？**

- 前端限制：浏览器单次 POST 有大小限制（通常 2GB）
- 网络不稳：传 10GB 文件中途断网，不想从头再来
- 进度显示：用户能看到"已传 35%，还剩 65%"

**分片上传流程**

```
┌─────────────┐    1. initChunkUpload     ┌─────────────┐
│   前端       │ ───────────────────────> │   后端       │
│             │ ←─────────────────────── │             │
│             │   返回 uploadId="abc123"  │             │
└─────────────┘                           └─────────────┘
         │                                        │
         │ 2. uploadChunk (循环调用)              │ 分片存到 temp/abc123/chunk_0
         │ ─────────────────────────────────────> │
         │ <───────────────────────────────────── │
         │       返回"已接收第0片"                 │
         │                                        │
         │ 3. getChunkProgress (可选)             │ 查询已传分片列表
         │ ─────────────────────────────────────> │
         │ <───────────────────────────────────── │
         │       返回[0,1,2] (已传3片)            │
         │                                        │
         │ 4. mergeChunks                         │ 按顺序合并所有分片 → 完整文件
         │ ─────────────────────────────────────> │ 写入 upload 目录，删 temp
         │ <───────────────────────────────────── │ 写入 file_info，返回 FileInfo
         │       返回文件信息                      │
         │                                        │
```

**关键设计点**

1. **分片大小固定**：前端统一切成 10MB 一片，后端不用管
2. **临时目录清理**：合并成功后自动删 temp 目录；也可定时任务清理 24 小时前的旧任务
3. **Redis 记录进度**：任务7目前用本地文件列表记录，任务14会改成 Redis，重启服务不丢进度

## 五、配置与扩展

### 5.1 配置文件

```yaml
# application.yml
cloud-disk:
  storage:
    root-path: D:\cloud-disk-files   # 存储根目录
    chunk-size: 10485760             # 分片大小（10MB）
    temp-expire-hours: 24            # 临时文件保留时间
```

### 5.2 策略模式预留

虽然现在只用本地磁盘，但代码结构已预留**存储策略模式**：

```java
// 接口
public interface IStorageService {
    String uploadFile(MultipartFile file, Long userId);
    FileInfo saveFileInfo(FileInfo fileInfo);
}

// 本地实现
@Service("localStorageService")
public class LocalStorageServiceImpl implements IStorageService { ... }

// 未来加 MinIO 实现
@Service("minioStorageService")
public class MinioStorageServiceImpl implements IStorageService { ... }
```

切换存储只需改一个注解：
```java
// @Autowired
// private IStorageService storageService;  // 默认

@Qualifier("minioStorageService")  // 切到 MinIO
private IStorageService storageService;
```

**为什么用策略模式？**

- 可插拔：明天想接阿里云 OSS，写个实现类就行
- 业务无感：FileServiceImpl 不知道文件存哪，只管调用接口
- 测试方便：Mock 一个 IStorageService 就能测业务逻辑

## 六、同方案对比

| 存储方案 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| **本地磁盘** | 零依赖、文件直接可见、速度最快 | 单机、硬盘坏了数据全丢 | 个人项目、内网环境 |
| **MinIO** | 兼容 S3、多节点、有版本控制 | 多一个进程、学习成本 | 小团队、需要对象存储特性 |
| **阿里云 OSS** | 无限扩容、CDN 加速、99.999999999% 可靠性 | 按流量收费、依赖外网 | 生产环境、对外服务 |
| **数据库 BLOB** | 数据统一管理、事务保证 | 性能差、数据库压力大 | 小文件（<1MB）、文档管理系统 |

**为什么个人项目选本地磁盘？**

1. 你只有一台电脑，不需要"多节点高可用"
2. 文件就在硬盘上，出问题直接 `cp -r` 备份
3. 不用为云存储付费（OSS 流量费不便宜）

## 七、安全考虑

1. **路径穿越防护**：`fileName` 参数过滤 `../`、`..\`、绝对路径
2. **文件类型检查**：前端限制 `.exe`、`.bat` 等可执行文件上传
3. **磁盘空间监控**：定时检查 `root-path` 剩余空间，低于 10% 告警
4. **病毒扫描**：可集成 ClamAV 扫描上传文件（任务16）

## 八、性能数据

| 操作 | 本地磁盘耗时 | 说明 |
|------|------------|------|
| 上传 100MB 文件 | ~200ms | 内存写入，取决于硬盘速度（SSD） |
| 分片合并 1GB | ~2s | 顺序读取 100 个分片，合并写入 |
| 列表 1000 个文件 | ~50ms | 数据库分页，不读磁盘 |

**瓶颈在哪？**

- 小文件（<10MB）：网络带宽（你家宽带 100Mbps ≈ 12MB/s）
- 大文件（>1GB）：硬盘 IO（SATA SSD 约 500MB/s）
- 分片合并：CPU（合并时要用 `FileInputStream` + `FileOutputStream` 循环）

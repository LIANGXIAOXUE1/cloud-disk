关键设计点：

1. uploadId：Redis生成唯一标识，关联分片信息，支持页面关闭后续传

2. 分片元数据：Redis缓存分片上传状态（chunkCount、uploadedChunks），合并后清除

3. MinIO临时存储：分片先存本地MinIO，合并成功后删除分片，释放空间

4. 存储分流：大文件（>100MB）自动上传阿里云盘，私密文件留本地MinIO

   6.3 策略调度上下文
   @Service
   public class StorageContext {
    @Autowired
    private Map<String, StorageStrategy> strategyMap;
    // 根据存储类型获取策略
    public StorageStrategy getStrategy(StorageType type) {
   
        return strategyMap.get(type.name());
   
    }
    // 存储选择算法（自动分流）
    public StorageType selectStorage(FileInfo fileInfo) {
   
        // 大文件 > 100MB → 阿里云盘（不限速）
        // 私密文件标记 → 本地MinIO
        // 小文件 → 就近原则/负载均衡
        // OneDrive教育版 → 空间最大优先
   
    }
   }

若接口返回 404/500：检查后端是否启动、接口路径是否正确、请求参数是否匹配
若接口返回 200 但前端无展示：检查前端渲染逻辑、响应数据解析是否正确
若样式错乱：切换到「Elements」标签，修改 CSS 样式，实时预览效果
3. 全流程调试（每个任务完成后必做）
例如：完成分片上传功能后，完整流程调试：
启动后端、前端、MySQL、Redis、RabbitMQ
登录前端账号，进入文件管理页
选择一个大文件（如 100MB），点击上传，选择存储仓库
查看前端上传进度，确认能正常上传
上传完成后，查看文件列表是否显示该文件，点击预览是否正常
尝试断网后重新上传，确认断点续传生效
若有问题，分别调试后端接口、前端组件，定位问题并修复
4. 调试注意事项（避坑关键）
每次只调试一个功能 / 接口，不要同时调试多个，避免混乱
遇到报错，先看控制台（IDEA 控制台、浏览器控制台），报错信息会明确提示问题所在（如 “数据库表不存在”“Token 无效”）
AI 生成的代码可能有小 bug（如参数错误、路径错误），调试时重点查看这些细节
调试完成后，记得 commit 代码（存档），避免调试崩了无法回退
若调试半天找不到问题，可重新让 OpenCode 生成该模块代码，对比差异，快速定位问题
六、VibeCoding+Git 配合技巧（高效开发，避免踩坑）
每个任务对应一个 Git 分支（命名格式：feature / 任务名称，如 feature/user-login）
每次启动任务前，先切回 main 分支，拉取最新代码，再开新分支
每个任务完成（调试通过）后，commit 一次，提交信息规范（如feat: 完成用户登录功能）
任务完成后，合并分支到 main，推送至 GitHub，再删除该任务分支
用 OpenCode 生成代码前，先贴 CLAUDE.md 规则，确保代码符合技术栈、目录结构要求
每次生成代码后，先调试，确认无问题再 commit，避免提交错误代码
